# 코딩테스트 체크리스트

풀 때마다 반복적으로 틀렸던 유형들을 정리합니다. 제출 전에 한 번씩 훑어보기.

## 인덱스 범위 처리 잘하기

- **길이가 다른 배열끼리 인덱스를 그대로 재사용하지 않기**: 예를 들어 "i번째와 i+1번째 사이의 값"을 담은 배열(길이 N-1)은, 현재 인덱스 `idx`가 아니라 `idx-1`로 접근해야 하는 경우가 많다 (ABC464 D에서 `G_H[idx]` 대신 `G_H[idx-1]`이어야 했던 것처럼).
- **반복문 종료 조건과 실제 접근하는 인덱스가 맞는지 다시 세보기**: `while idx < N: idx += 1; ...(idx 사용)...` 같은 구조는 마지막에 `idx == N`인 상태로 한 번 더 접근하게 되는 함정이 있다. `for idx in range(1, N):`처럼 범위가 명확한 형태를 우선 고려한다.
- **`[0] * N`과 `[0*N]`은 완전히 다르다**: 후자는 길이 1짜리 리스트다. 배열 초기화 직후 `len(...)`을 찍어서 의도한 크기가 맞는지 확인하는 습관을 들인다.
- **경계값(N=1, 빈 입력, 모든 원소가 같은 값 등) 손으로 한 번 넣어보기**: 배열 크기를 넉넉하게 잡았는지, 인덱스 상한이 실제 최댓값을 커버하는지 극단적인 입력으로 확인한다 (ABC459 C에서 N=1일 때 오프셋이 매 쿼리마다 증가해서 배열 크기를 두 배로 잡아야 했던 사례).

## 문제 잘 읽기

- **조건에 붙은 계수/부등호를 놓치지 않기**: "A 이하"인지 "A의 2배 이하"인지, "이상"인지 "초과"인지 — 이런 디테일 하나가 전체 로직을 바꾼다 (ABC460 C에서 `네타 ≤ 샤리`가 아니라 `네타 ≤ 샤리 × 2`였던 것, ABC463 C에서 `bisect_left`가 아니라 `bisect_right`를 써야 했던 것).
- **보너스/조건이 "선택한 결과"에 대한 것인지 "원래 입력값"에 대한 것인지 구분하기**: 최종 상태를 기준으로 판단해야 하는 조건을 원래 입력 기준으로 잘못 적용하면, 특정 분기에서만 조건을 빠뜨리는 실수가 생긴다 (ABC464 D에서 보너스 Y가 "오늘의 원래 날씨"와 무관하게 항상 적용돼야 했는데, 한쪽 분기에서만 빠졌던 사례).
- **입력 줄 순서를 문제 설명과 대조하기**: 변수 이름을 먼저 짓고 입력을 그 이름 순서대로 읽으면, 실제 입력 형식과 다르게 배정하는 실수가 생기기 쉽다. 입력 형식 설명을 보면서 어느 줄이 무엇인지 하나씩 확인한다.
